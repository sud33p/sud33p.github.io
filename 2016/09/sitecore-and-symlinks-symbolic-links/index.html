Sitecore and Symlinks (Symbolic Links)

September 3, 2016 in Sitecore

You must be wondering, what’s the relation here? There isn’t really, yet there could very well be one, if you want… You are probably aware that there are lots of ways to structure a sitecore (VS) solution. I swear if I get a dollar for every time I see a new way of structuring sitecore projects, I would be 100 dollars richer (yea, I don’t think there are more than 100 ways). That’s still a lot.

Ok, back to the topic. Recently, one of the projects that I worked on had a slightly different approach of structuring projects/folders in the solution, mainly the Website publish folder where the web project publishes into. One of the very popular ways to arrange your sitecore project is to keep your sitecore website files separate from the codebase. This approach has a lot of benefits as opposed to developing directly on top of your IIS website folder. If you are not familiar with this development technique, I suggest you read Sean Kearney’s post here.

Even though this project followed the technique of separating the sitecore website files from the code project, the sitecore website folder was still kept inside the solution folder, the source controlled area. Gitignores were used to prevent the folder content from getting tracked. This meant that the placement of the other two folders that are part of the sitecore installation (Data and Databases), were left to the dev’s discretion. Below is an example of such a structure.

The sitecore install folder is on the left, which normally contains of ‘Data, Databases and Website’ folders. The ‘Website’ folder has been moved to the right here. This isn’t so bad per se, sitting at a relative location to the solution is a benefit. But if you are working on multiple projects or someone new joins the team, this can get confusing. I was not a big fan. Can we for once get best of both worlds?

Symlinks to the rescue

Using Symlinks you can create a junction link on the sitecore install folder, pointing to the ‘Website’ folder on the Repo. And it will make it look like the ‘Website’ folder exists within the install folder with all its contents, even though it is just a reference to the actual ‘Website’ folder within the repo. Think of it as an advanced shortcut, similar to the Windows shortcut link (.lnk) but with greater power.

What are Symlinks anyway? Symlinks or Symbolic links, are basically advanced shortcuts. You can create symbolic links to individual files or folders, and then these will appear like they are stored in the folder with the symbolic link even though the symbolic link only points to their real location. Read more here

The command

Syntax: 


    MKLINK [[/D] | [/H] | [/J]] Link Target
        /D          Creates a directory symbolic link.  Default is a file symbolic link.
        /H          Creates a hard link instead of a symbolic link.
        /J          Creates a Directory Junction.
        Link        Specifies the new symbolic link name.
        Target      Specifies the path (relative or absolute) that the new link refers to.

For our purpose, we need to create a directory junction (/J) to link to the Website folder. So lets go ahead and do that:

  1. Open command prompt (in Admin mode)

  2. Enter the command mklink /J Link Target. In our case:

    D:\Dev\Sites\ClientA>mklink /J Website D:\Dev\Repos\ClientA\src\Website

    If successful, we will see the following output:

    Junction created for Website <<===>> D:\Dev\Repos\ClientA\src\Website

`MKLINK` is a built-in command line symlink tool in Windows Vista and above. If you are more of a GUI person, Link Shell Extension is an excellent free tool for you. It also supports Windows XP.

Result

We should now have a symlink named ‘Website’ created in the sitecore install folder which appears as a folder but is really pointing to the ‘Website’ folder in the repo. You can identify by the little shortcut icon on it.

Go ahead and open it. You should see all of the same website content listed underneath.

As a quick test, create a test file underneath either of the folder. You will see it appear immediately under the other.

All done. Now you can use this symlink reference as a real ‘Website’ folder for anything you want, be it for IIS site path, TDS project or grunt/gulp tasks.

One thing to remember is, even though these symlinks are a reference to the original item, they do not inherit permissions. So if you are going to use it for say, an IIS site, make sure to set appropriate permissions on the symlink as well.
