Sitecore's site definition name cannot start with a number

August 15, 2017 in Sitecore · 2 minute read

I felt the need for this post because first of all, this is most likely a rare scenario where you will need to name a site definition starting with a number. And when you do, the error you get will baffle you as it is in no way directly related to the change you’ve made. Let me explain!

This is a typical site definition entry:

<site name="website" enableTracking="true" virtualFolder="/" physicalFolder="/" 
rootPath="/sitecore/content" startItem="/home" database="web" domain="extranet" 
allowDebug="true" cacheHtml="true" htmlCacheSize="50MB" registryCacheSize="0" 
viewStateCacheSize="0" xslCacheSize="25MB" filteredItemsCacheSize="10MB" 
enablePreview="true" enableWebEdit="true" enableDebugger="true" 
disableClientData="false" cacheRenderingParameters="true" 
renderingParametersCacheSize="10MB" />

This is a site definition that has name starting with a number:

<site name="3littlekittens" enableTracking="true" ...

When you run the site, you will see an error like below:

System.Xml.XPath.XPathException: Expression must evaluate to a node-set

The Error

System.Xml.XPath.XPathException: Expression must evaluate to a node-set.

If you have made a bunch of other changes in addition to the site definition, then at first glance you won’t be able to relate this error to the site definition change. Even with a deeper look it’s not easy to figure out why exactly a simple name change would result in this XPath exception. The clue, as always (or in most cases), is in the stack trace.

Expression must evaluate to a node-set

You will need to decompile the above line to find out exactly why:

private long GetCacheSize(string attribute, Sitecore.Collections.StringDictionary properties, long defaultSize)
{
    ...
    string str = Factory.CreateObject("cacheSizes/sites/" + this.name + "/" + attribute.Substring(0, attribute.Length - "CacheSize".Length), false) as string;
    if (!string.IsNullOrEmpty(str))
        return StringUtil.ParseSizeString(str);
      return defaultSize;
}

Turns out that it’s not the <site> definition that is the real problem, but the corresponding entry in the <cacheSizes> definition that causes the error.

<cacheSizes>
    <sites>
      <website>
        <html>50MB</html>
        <registry>0</registry>
        <viewState>0</viewState>
        <xsl>25MB</xsl>
      </website>
      <3littlekittens>  <!-- this is invalid xml -->
        ...
      </3littlekittens>
    </sites>
  </cacheSizes>

The above code is trying to create an object out of the cacheSizes/sites/3littlekittens node (even if it’s not present), which would be an invalid xml as one of the rule states that it cannot start with a number. So now we know what is exactly behind this error.

Solution

Change the site name to not start with a number. :)

This is by far the simplest solution that you can apply in most scenarios as it’s just the site name. I haven’t had a need to look for a better solution yet. Hope this will work for you too.
